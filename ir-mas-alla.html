Indo Além: o Crew de Enriquecimento

Um lab de produção completo: fan-out/fan-in, fontes que falham com elegância e um contrato de saída estruturado

Por que enriquecer uma reclamação?

Os percursos principais do workshop são fluxos pré-configurados. Este lab é diferente: mostra como montar um crew de produção do zero, com todas as peças que fazem um crew escalar: múltiplas fontes de informação em paralelo, fontes que podem falhar sem derrubar a execução, e um contrato de saída estruturado que o próximo sistema pode consumir diretamente.

A tarefa escolhida é enriquecer uma reclamação/defeito de produto: receber um caso_id e um produto, e produzir um perfil completo com contexto web (recalls, normativa, avaliações), dados internos (histórico do caso no MongoDB) e um score determinístico de severidade/prioridade. É exatamente o tipo de pipeline que você encontra em produção: fan-out de pesquisa, fan-in de análise e uma saída validada pelo Pydantic.

O que você vai construir:
  • Um crew de 3 agentes pesquisadores + 1 analista consolidador
  • Busca web com firecrawl (citação de fontes rastreáveis)
  • Consulta interna com mongodb-vector-search em NL (sem SQL manual)
  • Uma fonte de dados regulatórios pública como terceiro braço paralelo
  • Um esquema Pydantic EnrichedDefectProfile como contrato de saída
  • Uma ferramenta determinística de score de severidade (Python puro)

Seção 1 — Dois agentes e um pipeline de tasks (~15 min)

Começamos com a versão mais simples possível: dois agentes e duas tasks encadeadas. O objetivo é executar algo e ler a saída com olho crítico.

Scaffolding do projeto

bash 
crewai create crew defect_enrichment
cd defect_enrichment
crewai install
Ponto de controle: crewai install termina sem erros em vermelho. Um erro de resolução de dependências aqui significa que a sua versão do CLI não coincide com o lab — revise o passo de instalação.

O crew.py inicial

python 
"""Defect enrichment crew: a web researcher and a consolidating analyst."""

from crewai import Agent, Crew, Process, Task
from crewai.project import CrewBase, agent, crew, task
from crewai.agents.agent_builder.base_agent import BaseAgent

@CrewBase
class DefectEnrichmentCrew():
    """Research a product defect, then consolidate into an actionable profile."""

    agents: list[BaseAgent]
    tasks: list[Task]

    @agent
    def investigador_web(self) -> Agent:
        return Agent(
            config=self.agents_config['investigador_web'],
            max_iter=8,
            allow_delegation=False,
            verbose=True,
        )

    @agent
    def analista_enriquecimiento(self) -> Agent:
        return Agent(
            config=self.agents_config['analista_enriquecimiento'],
            max_iter=6,
            allow_delegation=False,
            verbose=True,
        )

    @task
    def investigar_web(self) -> Task:
        return Task(
            config=self.tasks_config['investigar_web'],
        )

    @task
    def compilar_perfil(self) -> Task:
        return Task(
            config=self.tasks_config['compilar_perfil'],
            context=[self.investigar_web()],
        )

    @crew
    def crew(self) -> Crew:
        """Creates the DefectEnrichmentCrew"""
        return Crew(
            agents=self.agents,
            tasks=self.tasks,
            process=Process.sequential,
            verbose=True,
        )

# Ou com um caso específico:
uv run run_crew "REC-042" "Eletrônica - Carregador USB-C"

Agora olhe mais de perto — ele está mentindo

Abra o relatório e procure os detalhes: URLs de recalls, números de regulação, datas de alertas de segurança. As URLs são reais? Clique em uma. Provavelmente não leva a lugar nenhum, ou leva a uma página diferente da citada.

Lição-chave: Um agente sem ferramentas é um estagiário muito confiante, sem telefone nem internet. Inventa dados com a mesma confiança com que citaria dados reais. Tudo o que vem a seguir neste lab é sobre ancoragem: fontes reais por dentro, citações rastreáveis por fora.

Seção 2 — Dar ferramentas reais ao pesquisador (~20 min)

A solução é dar ao investigador_web ferramentas reais de busca e scraping. Usamos firecrawl: uma ferramenta que combina busca web e extração de conteúdo de páginas, disponível no AMP.

Instale o firecrawl

bash 
# A partir do diretório do projeto
crewai tool install firecrawl

# Ou com pip:
pip install crewai-tools[firecrawl]

Você precisa de uma FIRECRAWL_API_KEY no seu .env (gratuita em firecrawl.dev).

Conecte o firecrawl ao pesquisador

Em crew.py, importe e passe as ferramentas ao agente:

from crewai_tools import FirecrawlSearchTool, FirecrawlScrapeWebsiteTool

@agent
def investigador_web(self) -> Agent:
    return Agent(
        config=self.agents_config['investigador_web'],
        tools=[
            FirecrawlSearchTool(),       # buscar na web
            FirecrawlScrapeWebsiteTool(), # extrair conteúdo de URLs

        max_iter=8,
        max_rpm=10,
        allow_delegation=False,
        verbose=True,
    )

Seção 3 — O contrato de saída (~15 min)

Por enquanto o crew produz texto em Markdown. Está bem para leitura, mas se você quiser que outro sistema (uma API, um dashboard, um crew downstream) consuma a saída, você precisa de um contrato estruturado. No CrewAI, esse contrato é um esquema Pydantic em output_pydantic.

O esquema EnrichedDefectProfile

Crie src/defect_enrichment/schemas.py:

python 
"""Structured output schema for the defect enrichment crew."""

from pydantic import BaseModel, Field

class EnrichedDefectProfile(BaseModel):
    """The consolidated, decision-ready profile the crew produces."""

    caso_id: str = Field(description="Case ID, e.g. REC-042")
    produto: str = Field(description="Product name or description")
    descripcion_defecto: str = Field(
        description="One-paragraph description of the defect"
    )
    categoria: str = Field(description="Defect category, e.g. Segurança, Garantia")
    urgencia: str = Field(
        description="Urgency level: Alta, Media, Baja, or 'desconhecida'"
    )
    recalls_activos: list[str] = Field(
        default_factory=list,
        description="Active recalls or safety alerts related to the defect"
    )
    normativa_aplicable: list[str] = Field(
        default_factory=list,
        description="Applicable regulations or standards"
    )
    senales_recientes: list[str] = Field(
        default_factory=list,
        description="Recent signals: press, forums, regulatory filings"
    )
    resumen_interno: str = Field(
        default="Sem registros internos.",
        description="What internal systems returned (or why skipped)"
    )
    resumen_api_publica: str = Field(
        default="",
        description="Background from the public regulatory/consumer data API"
    )
    severity_score: int = Field(
        ge=0, le=100,
        description="0-100 deterministic severity/priority score"
    )
    severity_rationale: str = Field(
        description="Why this score — the reasoning behind the severity"
    )
    accion_recomendada: str = Field(
        description="One concrete next action for the resolution team"
    )
    sources: list[str] = Field(
        default_factory=list,
        description="Source URLs used in the research"
    )
    markdown_report: str = Field(
        description="A polished consolidated defect profile in Markdown, "
        "combining all findings into a readable brief."
    )

Ative o contrato em compilar_perfil

Em crew.py, importe o esquema e passe-o à task:

@task
def compilar_perfil(self) -> Task:

A armadilha mais comum: expected_output é sempre uma descrição em prosa. A classe Pydantic vai em output_pydantic, nunca dentro de expected_output. Misturá-los é a razão número um pela qual result.pydantic retorna None.

Atualize também o main.py para gravar os arquivos de saída:

python 
import sys
import warnings
from pathlib import Path

from defect_enrichment.crew import DefectEnrichmentCrew

warnings.filterwarnings("ignore", category=SyntaxWarning, module="pysbd")

DEFAULT_CASO = "REC-001"
DEFAULT_PRODUTO = "Eletrônica - Carregador USB-C"

def run():
    caso_id = sys.argv[1] if len(sys.argv) > 1 else DEFAULT_CASO
    produto = sys.argv[2] if len(sys.argv) > 2 else DEFAULT_PRODUTO

    out_dir = Path("output")
    out_dir.mkdir(exist_ok=True)

    inputs = {"caso_id": caso_id, "produto": produto}
    print(f"\nEnriquecendo: {caso_id} — {produto}\n" + "-" * 50)

    result = DefectEnrichmentCrew().crew().kickoff(inputs=inputs)

    if result.pydantic:
        profile = result.pydantic
        (out_dir / "defect_report.md").write_text(
            profile.markdown_report, encoding="utf-8"
        )
        (out_dir / "enriched_defect.json").write_text(
            profile.model_dump_json(indent=2), encoding="utf-8"
        )
        print("\n=== RESUMO ===")
        print(f"Categoria:  {profile.categoria}")
        print(f"Urgência:   {profile.urgencia}")
        print(f"Severidade: {profile.severity_score}/100")
        print(f"Ação:       {profile.accion_recomendada}")
        print("\nGravou output/defect_report.md e output/enriched_defect.json")
    else:
        (out_dir / "defect_report.md").write_text(result.raw, encoding="utf-8")
        print("\n" + result.raw)
Ponto de controle: a execução termina e aparece o resumo com Categoria, Urgência e Ação no terminal. Abra output/enriched_defect.json — cada campo do esquema, validado e tipado. O mesmo resultado é agora tanto um relatório legível como dados consumíveis por máquina.

A ferramenta de dados regulatórios públicos

Crie src/defect_enrichment/tools/regulatory_api_tool.py:

python 
"""Public regulatory/product-safety data tool.
Uses the free OpenFDA API (no key required) to look up product safety data.
"""

import json
import urllib.parse
import urllib.request

from crewai.tools import BaseTool
from pydantic import BaseModel, Field

_SEARCH_URL = (
    "https://api.fda.gov/device/recall.json"
    "?search=product_description:{q}&limit=3"
)
_HEADERS = {"User-Agent": "CrewAI-Workshop/1.0 (defect enrichment lab)"}

class RegulatoryInput(BaseModel):
    produto: str = Field(
        description="Product name or category to look up in regulatory data, "
        "e.g. 'USB charger' or 'carregador USB'"
    )

class RegulatoryAPITool(BaseTool):
    name: str = "Regulatory Safety Data API"
    description: str = (
        "Fetch public product safety and recall data from the OpenFDA API "
        "(free, no key required). Returns recall summaries and classification "
        "information. Use for neutral, citable regulatory background."
    )
    args_schema: type[BaseModel] = RegulatoryInput

    def _run(self, produto: str) -> str:
        try:
            q = urllib.parse.quote(produto[:60])
            url = _SEARCH_URL.format(q=q)
            request = urllib.request.Request(url, headers=_HEADERS)
            with urllib.request.urlopen(request, timeout=15) as resp:
                data = json.loads(resp.read().decode("utf-8"))
            results = data.get("results", [])
            if not results:
                return (
                    f"No public recall entries found for '{produto}' in OpenFDA. "
                    "Proceeding without public regulatory data."
                )
            lines = []
            for r in results[:3]:
                lines.append(
                    f"RECALL CLASS: {r.get('recalling_firm', 'unknown')}\n"
                    f"REASON: {r.get('reason_for_recall', '')[:200]}\n"
                    f"STATUS: {r.get('status', '')}\n"
                    f"SOURCE: https://api.fda.gov/device/recall.json"
                )
            return "\n\n".join(lines)
        except Exception as exc:
            return (
                f"Regulatory API unavailable ({exc}). "
                "Proceeding without public regulatory data."
            )

A troca da ferramenta interna: ANTES → DEPOIS

Esta é a mudança-chave em relação ao lab original. No lab do guia, internal_data_researcher usa uma função fábrica que decide em tempo de construção qual ferramenta retornar:

ANTES — lab original (db_tool.py)

python 
"""NL2SQL com fallback elegante.
Sem DATABASE_URL → retorna NoDatabaseTool.
"""
import os
from crewai.tools import BaseTool
from pydantic import BaseModel, Field

class NoDatabaseTool(BaseTool):
    name: str = "Internal Company Database"
    description: str = (
        "Look up internal records. If no DB "
        "configured, says so and the crew "
        "continues with public sources."
    )
    args_schema: type[BaseModel] = _DBQueryInput

    def _run(self, question: str) -> str:
        return (
            "No internal database is configured "
            "(DATABASE_URL is not set). Skipping "
            "internal records and proceeding "
            "with public sources."
        )

def build_internal_db_tool() -> BaseTool:
    db_uri = os.getenv("DATABASE_URL")
    if db_uri:
        from crewai_tools import NL2SQLTool
        return NL2SQLTool(db_uri=db_uri)
    return NoDatabaseTool()

# Em crew.py:
tools=[build_internal_db_tool()]

DEPOIS — com mongodb-vector-search

python 
"""Consulta interna com MongoDBVectorSearch.
Suporta NL nos modos factual / semantic /
hybrid / auto. Não requer DATABASE_URL
nem SQL manual. Se a ferramenta não estiver
disponível, retorna uma mensagem descritiva
em vez de lançar uma exceção.
"""
from mongodb_vector_search import (
    MongoDBVectorSearch,
)

# Em crew.py:
tools=[MongoDBVectorSearch()]
Por que esta troca? O lab original precisa de uma função fábrica porque a ferramenta de banco de dados pode ou não existir no ambiente do participante — se não há DATABASE_URL, retorna NoDatabaseTool, uma ferramenta vazia que diz "não há banco de dados" em vez de quebrar. Neste workshop, mongodb-vector-search substitui esse par inteiro: é a ferramenta real que você já instalou, lida com NL em quatro modos (factual, semantic, hybrid, auto) e se não encontra registros diz na sua resposta — retorna, nunca lança. Você não precisa mais da função fábrica nem do fallback manual.

O crew.py completo (Seção 4)

python 
"""Defect enrichment crew: three parallel source researchers + consolidating analyst."""

from crewai import Agent, Crew, Process, Task
from crewai.project import CrewBase, agent, crew, task
from crewai.agents.agent_builder.base_agent import BaseAgent
from crewai_tools import FirecrawlSearchTool, FirecrawlScrapeWebsiteTool
from mongodb_vector_search import MongoDBVectorSearch

from defect_enrichment.schemas import EnrichedDefectProfile
from defect_enrichment.tools.regulatory_api_tool import RegulatoryAPITool

@CrewBase
class DefectEnrichmentCrew():
    """Research a product defect across three sources in parallel, then consolidate."""

    agents: list[BaseAgent]
    tasks: list[Task]

    @agent
    def investigador_web(self) -> Agent:
        return Agent(
            config=self.agents_config['investigador_web'],
            tools=[
                FirecrawlSearchTool(),
                FirecrawlScrapeWebsiteTool(),
            ],
            max_iter=8,
            max_rpm=10,
            allow_delegation=False,
            verbose=True,
        )

    @agent
    def investigador_regulatorio(self) -> Agent:
        return Agent(
            config=self.agents_config['investigador_regulatorio'],
            tools=[RegulatoryAPITool()],
            max_iter=5,
            allow_delegation=False,
            verbose=True,
        )

    @agent
    def investigador_interno(self) -> Agent:
        return Agent(
            config=self.agents_config['investigador_interno'],
            tools=[MongoDBVectorSearch()],
            max_iter=5,
            allow_delegation=False,
            verbose=True,
        )

    @agent
    def analista_enriquecimiento(self) -> Agent:
        return Agent(
            config=self.agents_config['analista_enriquecimiento'],
            max_iter=6,
            allow_delegation=False,
            verbose=True,
        )

    @task
    def investigar_web(self) -> Task:
        return Task(config=self.tasks_config['investigar_web'])

    @task
    def investigar_regulatorio(self) -> Task:
        return Task(config=self.tasks_config['investigar_regulatorio'])

    @task
    def investigar_interno(self) -> Task:
        return Task(config=self.tasks_config['investigar_interno'])

    @task
    def compilar_perfil(self) -> Task:
        return Task(
            config=self.tasks_config['compilar_perfil'],
            context=[
                self.investigar_web(),
                self.investigar_regulatorio(),
                self.investigar_interno(),
            ],
            output_pydantic=EnrichedDefectProfile,
        )

    @crew
    def crew(self) -> Crew:
        return Crew(
            agents=self.agents,
            tasks=self.tasks,
            process=Process.sequential,
            verbose=True,
        )

Como funciona o fan-out / fan-in

  • As três tasks de pesquisa levam async_execution: true, por isso são executadas em paralelo mesmo que o processo seja sequencial.
  • compilar_perfil não é async: lista as três tasks em context=[…], o que o torna o ponto de sincronização — aguarda as três terminarem e suas saídas são os únicos dados que ele vê.
  • A última task de um crew nunca pode ser async — o crew precisa de um resultado final para retornar.
Degradação elegante — uma ferramenta retorna, nunca lança. Se a API regulatória não responder, RegulatoryAPITool._run() captura a exceção e retorna uma string descritiva. O pesquisador interno faz o mesmo se não há registros. O analista recebe todas as saídas igual — algumas mais vazias — e as consolida sem travar. É assim que um crew de produção trata as fontes opcionais: as ausências são reportadas, nunca há crash.
Ponto de controle: nos logs você verá os três pesquisadores intercalados (estão rodando ao mesmo tempo). compilar_perfil aguarda os três terminarem e consolida tudo no mesmo JSON validado de antes, agora com as três fontes.

Seção 5 — Trabalho determinístico → ferramenta, não prompt (~10 min)

O analista pode calcular um score de severidade com um prompt, mas você obteria resultados diferentes a cada execução. Quando a lógica é determinística — um conjunto de regras que produz sempre o mesmo número dado os mesmos inputs — ela pertence a uma ferramenta Python, não a um prompt.

A ferramenta de severidade (severity_tool.py)

Crie src/defect_enrichment/tools/severity_tool.py:

python 
"""Deterministic severity/priority scorer for product defects."""

from crewai.tools import BaseTool
from pydantic import BaseModel, Field

# Categories that automatically raise severity
HIGH_IMPACT_CATEGORIES = {
    "segurança", "safety", "incêndio", "fire", "eletrocução",
    "electrocution", "lesão", "injury", "morte", "death",
    "recall", "retirada", "toxicidade", "toxicity",
}

# Signals that indicate systemic or widespread issues
SYSTEMIC_KEYWORDS = {
    "massivo", "widespread", "multiple", "vários", "numerosas",
    "padrão", "pattern", "sistêmico", "systemic", "lote", "batch",
    "série", "series", "alerta", "alert", "emergência", "emergency",
}

class SeverityInput(BaseModel):
    categoria: str = Field(
        description="Defect category, e.g. 'Segurança', 'Garantia', 'Faturamento'"
    )
    urgencia: str = Field(
        description="Urgency level: 'Alta', 'Media', 'Baja', or 'desconhecida'"
    )
    senales_recientes: str = Field(
        description="Recent signals as free text (recalls, press, regulatory)"
    )

class SeverityScoreTool(BaseTool):
    name: str = "Severity Score"
    description: str = (
        "Calculate a 0-100 deterministic severity/priority score for a product "
        "defect, based on its category, urgency level, and recent signals. "
        "Returns the score and a short rationale. Use this to produce a "
        "consistent, auditable score — not an LLM estimate."
    )
    args_schema: type[BaseModel] = SeverityInput

    def _run(
        self, categoria: str, urgencia: str, senales_recientes: str
    ) -> str:
        score = 0
        reasons: list[str] = []

        # Category impact (0-40)
        cat_l = (categoria or "").lower()
        if any(k in cat_l for k in HIGH_IMPACT_CATEGORIES):
            score += 40
            reasons.append(f"Categoria de alto impacto ({categoria}): +40")
        else:
            score += 15
            reasons.append(f"Categoria padrão ({categoria}): +15")

        # Urgency (0-35)
        urg_l = (urgencia or "").lower()
        if "alta" in urg_l or "high" in urg_l:
            score += 35
            reasons.append("Urgência Alta: +35")
        elif "media" in urg_l or "medium" in urg_l:
            score += 20
            reasons.append("Urgência Média: +20")
        elif "baja" in urg_l or "low" in urg_l:
            score += 5
            reasons.append("Urgência Baixa: +5")
        else:
            score += 10
            reasons.append("Urgência desconhecida: +10")

        # Systemic signals (0-25)
        signals_l = (senales_recientes or "").lower()
        hits = sorted({kw for kw in SYSTEMIC_KEYWORDS if kw in signals_l})
        if hits:
            bonus = min(25, 8 + 4 * len(hits))
            score += bonus
            reasons.append(f"Sinais sistêmicos {hits}: +{bonus}")
        else:
            reasons.append("Sem sinais sistêmicos detectados: +0")

        score = max(0, min(100, score))
        return f"SEVERITY_SCORE={score} | RATIONALE: " + "; ".join(reasons)

Copiar uma ferramenta do AMP para o local

Durante o workshop você usou mongodb-vector-search e firecrawl diretamente do AMP através do Studio. Quando trabalha em um projeto de código local, essas mesmas ferramentas são instaladas com o CLI do CrewAI.

O fluxo completo — desde autenticar-se até instalar uma ferramenta publicada — são quatro comandos:

  • crewai login Autenticar-se na sua conta do AMP. Obrigatório antes de qualquer operação com ferramentas.
  • crewai tool create your-tool Criar uma nova ferramenta localmente a partir de um template. Gera a estrutura de arquivos no diretório atual.
  • crewai tool publish Publicar a ferramenta local no repositório do AMP. Uma vez publicada, você pode compartilhá-la com sua equipe ou torná-la pública.
  • crewai tool install your-tool Instalar uma ferramenta do AMP no projeto local. Isso é o que você usa para trazer mongodb-vector-search a um projeto de código próprio.
bash 
# 1. Autenticar-se (uma única vez por sessão)
crewai login

# 2. Instalar mongodb-vector-search do AMP no projeto local
crewai tool install mongodb-vector-search

# 3. Ou instalar firecrawl
crewai tool install firecrawl
O que o AMP gerencia por ferramenta? Quando você publica uma ferramenta no AMP, pode configurar: nome e descrição visível para outros usuários, visibilidade (privada para sua equipe ou pública), variáveis de ambiente obrigatórias (como FIRECRAWL_API_KEY), histórico de versões e downloads, e acesso por equipe e papéis. Isso significa que você pode publicar uma ferramenta interna, restringi-la à sua organização, declarar as credenciais que ela precisa, e o instalador obtém tudo automaticamente.

Uma vez instalada, você importa a ferramenta como qualquer outro pacote Python:

python 
from mongodb_vector_search import MongoDBVectorSearch
from crewai_tools import FirecrawlSearchTool

# Use igual ao deste lab
tools=[MongoDBVectorSearch(), FirecrawlSearchTool()]

Studio vs. código: No Studio você instala as ferramentas pelo painel Browse AMP com um clique. Em código local, crewai tool install faz o mesmo a partir do terminal — o mesmo catálogo, a mesma ferramenta.

Padrões de Flow

Quando a ordem dos passos importa, um Flow orquestra o trabalho com estado tipado e ramificações. Os decoradores @start, @listen e @router encadeiam e roteiam os passos. Quatro padrões cobrem a maioria dos casos:

  • Router (agência seletiva) — um classificador roteia cada pedido, de uma simples chamada de LLM a um crew completo. Ex.: triagem de tickets de suporte.
  • Loop de Revisão — o agente refina a saída até atingir um critério de qualidade. Ex.: migração de código, auditoria.
  • Processamento em Lote — muitos itens processados em paralelo (async) para alta vazão. Ex.: triagem de 100 reclamações de uma vez.
  • Conversacional (@persist()) — mantém o estado entre sessões, para chatbots com memória.

Regra geral: comece com um Crew; suba para um Flow quando precisar de ordem fixa, ramificação ou estado persistente.
