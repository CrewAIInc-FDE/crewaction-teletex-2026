Ir Más Allá: el Crew de Enriquecimiento

Un lab de producción completo: fan-out/fan-in, fuentes que fallan con gracia y un contrato de salida estructurado

¿Por qué enriquecer un reclamo?

Los tres casos de uso del taller son flujos preconfigurados. Este lab es diferente: te muestra cómo armar un crew de producción desde cero, con todas las piezas que hacen que un crew escale: múltiples fuentes de información en paralelo, fuentes que pueden fallar sin tirar abajo la ejecución, y un contrato de salida estructurado que el siguiente sistema puede consumir directamente.

La tarea elegida es enriquecer un reclamo/defecto de producto: recibir un caso_id y un producto, y producir un perfil completo con contexto web (recalls, normativa, reseñas), datos internos (historial del caso en Teradata) y un score determinístico de severidad/prioridad. Es exactamente el tipo de pipeline que encontrás en producción: fan-out de investigación, fan-in de análisis y una salida validada por Pydantic.

Lo que vas a construir:
  • Un crew de 3 agentes investigadores + 1 analista consolidador
  • Búsqueda web con firecrawl (citación de fuentes trazables)
  • Consulta interna con teradata-store-search en NL (sin SQL manual)
  • Una fuente de datos regulatorios pública como tercer brazo paralelo
  • Un esquema Pydantic EnrichedDefectProfile como contrato de salida
  • Una herramienta determinística de score de severidad (Python puro)

Sección 1 — Dos agentes y un pipeline de tasks (~15 min)

Empezamos con la versión más simple posible: dos agentes y dos tasks encadenadas. El objetivo es correr algo y leer la salida con ojo crítico.

bash

cd defect_enrichment
crewai install

El crew.py inicial

python 
"""Defect enrichment crew: a web researcher and a consolidating analyst."""

from crewai import Agent, Crew, Process, Task
from crewai.project import CrewBase, agent, crew, task
from crewai.agents.agent_builder.base_agent import BaseAgent

@CrewBase
class DefectEnrichmentCrew():
    """Research a product defect, then consolidate into an actionable profile."""

    agents: list[BaseAgent]
    tasks: list[Task]

    @agent
    def investigador_web(self) -> Agent:
        return Agent(
            config=self.agents_config['investigador_web'],
            max_iter=8,
            allow_delegation=False,
            verbose=True,
        )

    @agent
    def analista_enriquecimiento(self) -> Agent:
        return Agent(
            config=self.agents_config['analista_enriquecimiento'],
            max_iter=6,
            allow_delegation=False,
            verbose=True,
        )

    @task
    def investigar_web(self) -> Task:
        return Task(
            config=self.tasks_config['investigar_web'],
        )

    @task
    def compilar_perfil(self) -> Task:
        return Task(
            config=self.tasks_config['compilar_perfil'],
            context=[self.investigar_web()],
        )

    @crew
    def crew(self) -> Crew:
        """Creates the DefectEnrichmentCrew"""
        return Crew(
            agents=self.agents,
            tasks=self.tasks,
            process=Process.sequential,
            verbose=True,
        )

Sección 3 — El contrato de salida (~15 min)

Por ahora el crew produce texto en Markdown. Está bien para leer, pero si querés que otro sistema (una API, un dashboard, un crew downstream) consuma la salida, necesitás un contrato estructurado. En CrewAI, ese contrato es un esquema Pydantic en output_pydantic.

El esquema EnrichedDefectProfile

Creá src/defect_enrichment/schemas.py:

python 
"""Structured output schema for the defect enrichment crew."""

from pydantic import BaseModel, Field

class EnrichedDefectProfile(BaseModel):
    """The consolidated, decision-ready profile the crew produces."""

    caso_id: str = Field(description="Case ID, e.g. REC-042")
    producto: str = Field(description="Product name or description")
    descripcion_defecto: str = Field(
        description="One-paragraph description of the defect"
    )
    categoria: str = Field(description="Defect category, e.g. Seguridad, Garantía")
    urgencia: str = Field(
        description="Urgency level: Alta, Media, Baja, or 'desconocida'"
    )
    recalls_activos: list[str] = Field(
        default_factory=list,
        description="Active recalls or safety alerts related to the defect"
    )
    normativa_aplicable: list[str] = Field(
        default_factory=list,
        description="Applicable regulations or standards"
    )
    senales_recientes: list[str] = Field(
        default_factory=list,
        description="Recent signals: press, forums, regulatory filings"
    )
    resumen_interno: str = Field(
        default="Sin registros internos.",
        description="What internal systems returned (or why skipped)"
    )
    resumen_api_publica: str = Field(
        default="",
        description="Background from the public regulatory/consumer data API"
    )
    severity_score: int = Field(
        ge=0, le=100,
        description="0-100 deterministic severity/priority score"
    )
    severity_rationale: str = Field(
        description="Why this score — the reasoning behind the severity"
    )
    accion_recomendada: str = Field(
        description="One concrete next action for the resolution team"
    )
    sources: list[str] = Field(
        default_factory=list,
        description="Source URLs used in the research"
    )
    markdown_report: str = Field(
        description="A polished consolidated defect profile in Markdown, "
        "combining all findings into a readable brief."
    )

Activá el contrato en compilar_perfil

En crew.py, importá el esquema y pasalo a la task:

La trampa más común: expected_output es siempre una descripción en prosa. La clase Pydantic va en output_pydantic, nunca dentro de expected_output. Mezclarlos es la razón número uno por la que result.pydantic vuelve None.

También actualizá main.py para escribir los archivos de salida:

python 
import sys
import warnings
from pathlib import Path

from defect_enrichment.crew import DefectEnrichmentCrew

warnings.filterwarnings("ignore", category=SyntaxWarning, module="pysbd")

DEFAULT_CASO = "REC-001"
DEFAULT_PRODUCTO = "Electrónica - Cargador USB-C"

def run():
    caso_id = sys.argv[1] if len(sys.argv) > 1 else DEFAULT_CASO
    producto = sys.argv[2] if len(sys.argv) > 2 else DEFAULT_PRODUCTO

    out_dir = Path("output")
    out_dir.mkdir(exist_ok=True)

    inputs = {"caso_id": caso_id, "producto": producto}
    print(f"\nEnriqueciendo: {caso_id} — {producto}\n" + "-" * 50)

    result = DefectEnrichmentCrew().crew().kickoff(inputs=inputs)

    if result.pydantic:
        profile = result.pydantic
        (out_dir / "defect_report.md").write_text(
            profile.markdown_report, encoding="utf-8"
        )
        (out_dir / "enriched_defect.json").write_text(
            profile.model_dump_json(indent=2), encoding="utf-8"
        )
        print("\n=== RESUMEN ===")
        print(f"Categoría:  {profile.categoria}")
        print(f"Urgencia:   {profile.urgencia}")
        print(f"Severidad:  {profile.severity_score}/100")
        print(f"Acción:     {profile.accion_recomendada}")
        print("\nEscribió output/defect_report.md y output/enriched_defect.json")
    else:
        (out_dir / "defect_report.md").write_text(result.raw, encoding="utf-8")
        print("\n" + result.raw)
Punto de control: la ejecución termina y aparece el resumen con Categoría, Urgencia y Acción en la terminal. Abrí output/enriched_defect.json — cada campo del esquema, validado y tipado. El mismo resultado es ahora tanto un informe legible como datos consumibles por máquina.

La herramienta de datos regulatorios públicos

Creá src/defect_enrichment/tools/regulatory_api_tool.py:

python 
"""Public regulatory/product-safety data tool.
Uses the free OpenFDA API (no key required) to look up product safety data.
"""

import json
import urllib.parse
import urllib.request

from crewai.tools import BaseTool
from pydantic import BaseModel, Field

_SEARCH_URL = (
    "https://api.fda.gov/device/recall.json"
    "?search=product_description:{q}&limit=3"
)
_HEADERS = {"User-Agent": "CrewAI-Workshop/1.0 (defect enrichment lab)"}

class RegulatoryInput(BaseModel):
    producto: str = Field(
        description="Product name or category to look up in regulatory data, "
        "e.g. 'USB charger' or 'cargador USB'"
    )

class RegulatoryAPITool(BaseTool):
    name: str = "Regulatory Safety Data API"
    description: str = (
        "Fetch public product safety and recall data from the OpenFDA API "
        "(free, no key required). Returns recall summaries and classification "
        "information. Use for neutral, citable regulatory background."
    )
    args_schema: type[BaseModel] = RegulatoryInput

    def _run(self, producto: str) -> str:
        try:
            q = urllib.parse.quote(producto[:60])
            url = _SEARCH_URL.format(q=q)
            request = urllib.request.Request(url, headers=_HEADERS)
            with urllib.request.urlopen(request, timeout=15) as resp:
                data = json.loads(resp.read().decode("utf-8"))
            results = data.get("results", [])
            if not results:
                return (
                    f"No public recall entries found for '{producto}' in OpenFDA. "
                    "Proceeding without public regulatory data."
                )
            lines = []
            for r in results[:3]:
                lines.append(
                    f"RECALL CLASS: {r.get('recalling_firm', 'unknown')}\n"
                    f"REASON: {r.get('reason_for_recall', '')[:200]}\n"
                    f"STATUS: {r.get('status', '')}\n"
                    f"SOURCE: https://api.fda.gov/device/recall.json"
                )
            return "\n\n".join(lines)
        except Exception as exc:
            return (
                f"Regulatory API unavailable ({exc}). "
                "Proceeding without public regulatory data."
            )

El swap de la herramienta interna: ANTES → DESPUÉS

Este es el cambio clave respecto al lab original. En el lab de la guía, internal_data_researcher usa una función fábrica que decide en tiempo de construcción qué herramienta retornar:

ANTES — lab original (db_tool.py)

python 
"""NL2SQL con fallback graceful.
Sin DATABASE_URL → retorna NoDatabaseTool.
"""
import os
from crewai.tools import BaseTool
from pydantic import BaseModel, Field

class NoDatabaseTool(BaseTool):
    name: str = "Internal Company Database"
    description: str = (
        "Look up internal records. If no DB "
        "configured, says so and the crew "
        "continues with public sources."
    )
    args_schema: type[BaseModel] = _DBQueryInput

    def _run(self, question: str) -> str:
        return (
            "No internal database is configured "
            "(DATABASE_URL is not set). Skipping "
            "internal records and proceeding "
            "with public sources."
        )

def build_internal_db_tool() -> BaseTool:
    db_uri = os.getenv("DATABASE_URL")
    if db_uri:
        from crewai_tools import NL2SQLTool
        return NL2SQLTool(db_uri=db_uri)
    return NoDatabaseTool()

# En crew.py:
tools=[build_internal_db_tool()]

DESPUÉS — con teradata-store-search

python 
"""Consulta interna con TeradataStoreSearch.
Soporta NL en modos factual / semantic /
hybrid / auto. No requiere DATABASE_URL
ni SQL manual. Si la herramienta no está
disponible, retorna un mensaje descriptivo
en lugar de lanzar una excepción.
"""
from teradata_store_search import (
    TeradataStoreSearch,
)

# En crew.py:
tools=[TeradataStoreSearch()]
¿Por qué este swap? El lab original necesita una función fábrica porque la herramienta de base de datos puede o no existir en el entorno del participante — si no hay DATABASE_URL, retorna NoDatabaseTool, una herramienta vacía que dice "no hay base de datos" en lugar de romper. En este taller, teradata-store-search reemplaza ese par completo: es la herramienta real que ya instalaste, maneja NL en cuatro modos (factual, semantic, hybrid, auto) y si no encuentra registros lo dice en su respuesta — devuelve, nunca lanza. Ya no necesitás la función fábrica ni el fallback manual.

El crew.py completo (Sección 4)

python 
"""Defect enrichment crew: three parallel source researchers + consolidating analyst."""

from crewai import Agent, Crew, Process, Task
from crewai.project import CrewBase, agent, crew, task
from crewai.agents.agent_builder.base_agent import BaseAgent
from crewai_tools import FirecrawlSearchTool, FirecrawlScrapeWebsiteTool
from teradata_store_search import TeradataStoreSearch

from defect_enrichment.schemas import EnrichedDefectProfile
from defect_enrichment.tools.regulatory_api_tool import RegulatoryAPITool

@CrewBase
class DefectEnrichmentCrew():
    """Research a product defect across three sources in parallel, then consolidate."""

    agents: list[BaseAgent]
    tasks: list[Task]

    @agent
    def investigador_web(self) -> Agent:
        return Agent(
            config=self.agents_config['investigador_web'],
            tools=[
                FirecrawlSearchTool(),
                FirecrawlScrapeWebsiteTool(),
            ],
            max_iter=8,
            max_rpm=10,
            allow_delegation=False,
            verbose=True,
        )

    @agent
    def investigador_regulatorio(self) -> Agent:
        return Agent(
            config=self.agents_config['investigador_regulatorio'],
            tools=[RegulatoryAPITool()],
            max_iter=5,
            allow_delegation=False,
            verbose=True,
        )

    @agent
    def investigador_interno(self) -> Agent:
        return Agent(
            config=self.agents_config['investigador_interno'],
            tools=[TeradataStoreSearch()],
            max_iter=5,
            allow_delegation=False,
            verbose=True,
        )

    @agent
    def analista_enriquecimiento(self) -> Agent:
        return Agent(
            config=self.agents_config['analista_enriquecimiento'],
            max_iter=6,
            allow_delegation=False,
            verbose=True,
        )

    @task
    def investigar_web(self) -> Task:
        return Task(config=self.tasks_config['investigar_web'])

    @task
    def investigar_regulatorio(self) -> Task:
        return Task(config=self.tasks_config['investigar_regulatorio'])

    @task
    def investigar_interno(self) -> Task:
        return Task(config=self.tasks_config['investigar_interno'])

    @task
    def compilar_perfil(self) -> Task:
        return Task(
            config=self.tasks_config['compilar_perfil'],
            context=[
                self.investigar_web(),
                self.investigar_regulatorio(),
                self.investigar_interno(),
            ],
            output_pydantic=EnrichedDefectProfile,
        )

    @crew
    def crew(self) -> Crew:
        return Crew(
            agents=self.agents,
            tasks=self.tasks,
            process=Process.sequential,
            verbose=True,
        )

Cómo funciona el fan-out / fan-in

  • Las tres tasks de investigación llevan async_execution: true, por eso se ejecutan en paralelo aunque el proceso sea secuencial.
  • compilar_perfil no es async: lista las tres tasks en context=[…], lo que lo convierte en el punto de sincronización — espera a que terminen las tres y sus salidas son los únicos datos que ve.
  • La última task de un crew nunca puede ser async — el crew necesita un resultado final para retornar.
Degradación elegante — una herramienta retorna, nunca lanza. Si la API regulatoria no responde, RegulatoryAPITool._run() captura la excepción y retorna un string descriptivo. El investigador interno hace lo mismo si no hay registros. El analista recibe todos los outputs igual — algunos más vacíos — y los consolida sin crashear. Así es como un crew de producción trata las fuentes opcionales: las ausencias se reportan, nunca se crashea.
Punto de control: en los logs vas a ver los tres investigadores intercalados (están corriendo al mismo tiempo). compilar_perfil espera a que los tres terminen y consolida todo en el mismo JSON validado de antes, ahora con las tres fuentes.

Sección 5 — Trabajo determinístico → herramienta, no prompt (~10 min)

El analista puede calcular un score de severidad con un prompt, pero obtendrías resultados distintos en cada ejecución. Cuando la lógica es determinística — un conjunto de reglas que produce siempre el mismo número dado los mismos inputs — pertenece a una herramienta Python, no a un prompt.

La herramienta de severidad (severity_tool.py)

Creá src/defect_enrichment/tools/severity_tool.py:

python 
"""Deterministic severity/priority scorer for product defects."""

from crewai.tools import BaseTool
from pydantic import BaseModel, Field

# Categories that automatically raise severity
HIGH_IMPACT_CATEGORIES = {
    "seguridad", "safety", "incendio", "fire", "electrocucion",
    "electrocution", "lesion", "injury", "muerte", "death",
    "recall", "retiro", "toxicidad", "toxicity",
}

# Signals that indicate systemic or widespread issues
SYSTEMIC_KEYWORDS = {
    "masivo", "widespread", "multiple", "varios", "numerosas",
    "patrón", "pattern", "sistémico", "systemic", "lote", "batch",
    "serie", "series", "alerta", "alert", "emergencia", "emergency",
}

class SeverityInput(BaseModel):
    categoria: str = Field(
        description="Defect category, e.g. 'Seguridad', 'Garantía', 'Facturación'"
    )
    urgencia: str = Field(
        description="Urgency level: 'Alta', 'Media', 'Baja', or 'desconocida'"
    )
    senales_recientes: str = Field(
        description="Recent signals as free text (recalls, press, regulatory)"
    )

class SeverityScoreTool(BaseTool):
    name: str = "Severity Score"
    description: str = (
        "Calculate a 0-100 deterministic severity/priority score for a product "
        "defect, based on its category, urgency level, and recent signals. "
        "Returns the score and a short rationale. Use this to produce a "
        "consistent, auditable score — not an LLM estimate."
    )
    args_schema: type[BaseModel] = SeverityInput

    def _run(
        self, categoria: str, urgencia: str, senales_recientes: str
    ) -> str:
        score = 0
        reasons: list[str] = []

        # Category impact (0-40)
        cat_l = (categoria or "").lower()
        if any(k in cat_l for k in HIGH_IMPACT_CATEGORIES):
            score += 40
            reasons.append(f"Categoría de alto impacto ({categoria}): +40")
        else:
            score += 15
            reasons.append(f"Categoría estándar ({categoria}): +15")

        # Urgency (0-35)
        urg_l = (urgencia or "").lower()
        if "alta" in urg_l or "high" in urg_l:
            score += 35
            reasons.append("Urgencia Alta: +35")
        elif "media" in urg_l or "medium" in urg_l:
            score += 20
            reasons.append("Urgencia Media: +20")
        elif "baja" in urg_l or "low" in urg_l:
            score += 5
            reasons.append("Urgencia Baja: +5")
        else:
            score += 10
            reasons.append("Urgencia desconocida: +10")

        # Systemic signals (0-25)
        signals_l = (senales_recientes or "").lower()
        hits = sorted({kw for kw in SYSTEMIC_KEYWORDS if kw in signals_l})
        if hits:
            bonus = min(25, 8 + 4 * len(hits))
            score += bonus
            reasons.append(f"Señales sistémicas {hits}: +{bonus}")
        else:
            reasons.append("Sin señales sistémicas detectadas: +0")

        score = max(0, min(100, score))
        return f"SEVERITY_SCORE={score} | RATIONALE: " + "; ".join(reasons)

Copiar una herramienta de AMP a local

Durante el taller usaste teradata-store-search y firecrawl directamente desde AMP a través de Studio. Cuando trabajás en un proyecto de código local, esas mismas herramientas se instalan con el CLI de CrewAI.

El flujo completo — desde autenticarte hasta instalar una herramienta publicada — son cuatro comandos:

  • crewai login Autenticarte en tu cuenta de AMP. Requerido antes de cualquier operación con herramientas.
  • crewai tool create your-tool Crear una nueva herramienta localmente desde un template. Genera la estructura de archivos en el directorio actual.
  • crewai tool publish Publicar la herramienta local al repositorio de AMP. Una vez publicada, podés compartirla con tu equipo o hacerla pública.
  • crewai tool install your-tool Instalar una herramienta desde AMP al proyecto local. Esto es lo que usás para traer teradata-store-search a un proyecto de código propio.
bash 
# 1. Autenticarse (una sola vez por sesión)
crewai login

# 2. Instalar teradata-store-search desde AMP al proyecto local
crewai tool install teradata-store-search

# 3. O instalar firecrawl
crewai tool install firecrawl
¿Qué gestiona AMP por herramienta? Cuando publicás una herramienta en AMP, podés configurar: nombre y descripción visible para otros usuarios, visibilidad (privada para tu equipo o pública), variables de entorno requeridas (como FIRECRAWL_API_KEY), historial de versiones y descargas, y acceso por equipo y roles. Esto significa que podés publicar una herramienta interna, restringirla a tu organización, declarar las credenciales que necesita, y el instalador lo obtiene todo automáticamente.

Una vez instalada, importás la herramienta igual que cualquier otro paquete Python:

python 
from teradata_store_search import TeradataStoreSearch
from crewai_tools import FirecrawlSearchTool

# Usala igual que en este lab
tools=[TeradataStoreSearch(), FirecrawlSearchTool()]

Studio vs. código: En Studio instalás las herramientas desde el panel Browse AMP con un click. En código local, crewai tool install hace lo mismo desde la terminal — el mismo catálogo, la misma herramienta.
